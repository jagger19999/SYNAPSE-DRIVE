# 90 分钟多模态采集数据吞吐说明

## 一、数据量估算（90 分钟）

| 数据流   | 采样率 | 90 分钟样本数 | 约 CSV 体积（估算） |
|----------|--------|----------------|----------------------|
| 驾驶数据 | 150 Hz | 810,000 行     | ~200–400 MB          |
| 生理数据 | 1000 Hz| 5,400,000 行   | ~1–1.5 GB           |
| 眼动数据 | 50 Hz  | 270,000 行     | ~50–100 MB           |
| **合计** | 1200 行/秒 | **~648 万行** | **~1.5–2 GB**       |

- 总写入速率约 **1200 行/秒**，对 SSD/HDD 来说通常没问题。
- 可能成为瓶颈的是：**队列缓冲不足**、**主线程/写入线程偶尔卡顿**，导致队列满而丢数。

---

## 二、当前架构与潜在风险

1. **生理 1000 Hz**
   - LSL `pull_chunk(max_samples=1000)` → `biosig_batch_queue`（maxsize=**1000**）→ 每 50 ms 由 `_process_biosig_batch` 取出 → `_biosig_stream_write_queue`（maxsize=50000）→ 独立写入线程写 CSV。
   - 风险：`biosig_batch_queue` 只有 **1000 条 ≈ 1 秒**缓冲。若主线程因 UI/其他逻辑卡住超过约 1 秒，会 `queue.Full` 静默丢包。
   - 写入队列 50000 条 ≈ 50 秒缓冲，正常写盘速度下够用，但在极端 I/O 或系统负载下仍可能积压。

2. **驾驶 150 Hz、眼动 50 Hz**
   - 各自有独立队列和写入线程，速率相对低，压力小于生理流。驾驶队列 20000、眼动 20000，对 90 分钟一般足够。

3. **内存**
   - 已有 `soft_cache_limit` / `hard_cache_limit`、定期 flush、UI 清理、约 5 分钟一次 GC，对 90 分钟连续跑是有考虑的。

---

## 三、建议的代码调整（降低 90 分钟丢数风险）

在不改变整体架构的前提下，建议只做**缓冲与 I/O 频率**的保守优化：

1. **增大生理中间队列**（防止主线程短时卡顿导致丢包）  
   - 在 LSL 接收线程里，把 `biosig_batch_queue = queue.Queue(maxsize=1000)` 改为 **5000 或 10000**（约 5–10 秒缓冲）。

2. **（可选）增大生理写入队列**  
   - `_biosig_stream_write_queue = queue.Queue(maxsize=50000)` 改为 **100000**，给写盘更多缓冲。

3. **（可选）降低生理 CSV 的 flush 频率**  
   - 生理写入线程里，把 `flush_counter >= 200` 改为 **500 或 1000**，减少 `file.flush()` 次数，降低 I/O 抖动（数据仍会先进入 OS 缓冲，风险可控）。

4. **保持或略减批处理间隔**  
   - `queue_process_interval = 50`（50 ms）已合理，可保持；若主线程负载高，可尝试改为 **30**，加快从 `biosig_batch_queue` 取数。

5. **90 分钟前检查**  
   - 磁盘剩余空间 ≥ **3 GB**（含系统与其它文件更稳妥）。  
   - 实验前做一次短时（如 1–2 分钟）三流同采，确认无队列满、无丢包或报错，再跑 90 分钟。

按上述调整后，**不会改变采样率或数据内容**，只是让 150 Hz 驾驶 + 1000 Hz 生理 + 50 Hz 眼动在 90 分钟下更不容易因短时阻塞或 I/O 波动而丢数。
